# RM 운영 이상탐지 1차 명세

## 목적
운영 담당자의 상담·스케줄링 판단을 대체하지 않고, 라이언이 원자료를 직접 순회하며 누락과 불일치를 찾는 작업을 줄인다.

## 기본 원칙
- 모든 검사는 읽기 전용으로 시작한다.
- 자동수정은 별도 승인 전까지 금지한다.
- 개인정보는 보고서와 로그에서 최소화한다.
- 탐지 결과에는 규칙 ID, 근거 행, 심각도, 권장 조치를 남긴다.
- 재실행 시 동일 이슈가 중복 생성되지 않도록 fingerprint를 사용한다.

## 규칙

### RM-DET-001 재등록 상태 누락
- 조건: 활성 학생, 잔여 수업 0~3회, 재등록 상태 공란
- 심각도: 검토
- 권장 조치: 담당 강사 확인 목록에 추가

### RM-DET-002 활성 학생 장기 미수강
- 조건: 활성 상태이며 마지막 수업 후 기준일 이상 경과
- 기본 기준일: 21일
- 심각도: 검토
- 예외: 연기, 휴학, 환불 진행, 사전 등록된 장기 출장

### RM-DET-003 향후 일정 없음
- 조건: 활성 상태이나 미래 수업 일정 없음
- 심각도: 검토
- 예외: 잔여 0회, 연기, 졸업, 환불 진행

### RM-DET-004 입금·패키지 불일치
- 조건 A: 입금 확인, 등록 패키지 없음
- 조건 B: 등록 패키지 생성, 입금 확인 없음
- 심각도: 긴급 또는 검토

### RM-DET-005 중복 수업
- 조건: 학생, 강사, 수업 시작일시가 동일한 기록 2건 이상
- 심각도: 긴급

### RM-DET-006 허용되지 않은 코드
- 조건: 수업 또는 정산 코드가 허용 목록 밖
- 알려진 값: 정상 수업, N/n(0회), S1(당일취소·정산 규칙은 별도 단가표 참조)
- 심각도: 긴급

### RM-DET-007 수업건수·정산시간 불일치
- 조건: 원자료의 정산 코드로 계산한 시간과 집계값 불일치
- 심각도: 긴급

### RM-DET-008 이름 표기 불일치
- 조건: 학생DB 또는 강사DB의 기준명과 다른 공백·대소문자·철자 변형
- 심각도: 검토
- 자동수정 후보일 수 있으나 초기에 제안만 생성

## 결과 스키마
- detection_id
- rule_id
- severity: urgent | review | suggestion
- entity_type
- entity_key_masked
- source_sheet
- source_row
- detected_at
- fingerprint
- evidence
- recommended_action
- status: open | acknowledged | resolved | ignored

## 완료 기준
- 정상, 빈값, 중복, 형식 오류, 예외, 재실행 테스트 포함
- 운영 데이터 쓰기 없음
- 동일 fingerprint 중복 보고 방지
- 오탐 조정을 위한 기준값 설정 분리

## 실데이터 교차원천 규칙

### RM-X-007 졸업로그 학생명 열의 강사명 입력
- 조건: 졸업로그의 학생명 값이 알려진 강사명 레이블과 일치
- 심각도: 긴급
- 자동판정 금지 사유: 원본문장 안의 실제 학생명을 사람이 확인해야 하며 강사명만으로 정정 대상을 확정할 수 없음

### RM-X-008 자동감지 알림과 확정 졸업 혼재
- 조건: 졸업로그 원문에 자동감지 표시가 있거나 상태가 알림전송
- 심각도: 검토
- 자동판정 금지 사유: 알림 후보는 확정 졸업 상태와 다른 업무 단계이므로 상태 전이를 사람이 확인해야 함

### RM-X-009 동일 학생·등록일·금액·회차 중복 등록
- 조건: 패키지 등록 후보 중 학생명, 등록일, 금액, 회차가 동일한 행이 2건 이상
- 심각도: 긴급
- 제외: 취소 거래, 교재비, 차액, 추가결제, 회차수 영향 없음으로 분류된 비패키지 거래
- 자동판정 금지 사유: 실제 중복인지 분할 결제/재처리인지 원문과 입금 로그 확인이 필요함

### RM-X-010 동일 졸업사유 분할·중복 후보
- 조건: 동일 학생, 동일 마지막수업일 또는 처리일, 동일 정규화 사유가 2건 이상
- 심각도: 검토
- 자동판정 금지 사유: 운영자가 의도적으로 사유를 분할 기록했을 수 있어 자동 병합하지 않음

### RM-X-011 학생-입금자 관계 검토
- 조건: 입금자명이 학생명과 다르거나 가족·법인·괄호 별칭·이름순서 변형·동명이인·전화번호 충돌 근거가 있는 경우
- 심각도: 동명이인 또는 식별자 충돌은 긴급, 그 외 alias 관계는 검토
- 보고: `confidence`와 `reasons`를 근거 문구에 포함
- 자동판정 금지 사유: 이름 단독, 가족입금, 법인입금, 괄호 표기는 실제 수강생 확정 근거가 아니므로 student_id 또는 전화번호 근거 확인 전 자동 연결하지 않음

## 학생-입금자 안전 매칭

1. `student_id` exact를 최우선으로 사용하며 충돌 시 conflict로 보고한다.
2. 정규화 전화번호 exact를 높은 신뢰 근거로 사용하며 번호 충돌은 conflict로 보고한다.
3. 괄호 안 별칭, 가족입금, 법인입금, 이름순서 변형은 review confidence를 유지한다.
4. 이름만 같은 경우 certain 판정을 금지하고 동명이인 후보가 2건 이상이면 conflict로 보고한다.
5. 모든 conflict/review 결과는 자동 수정하지 않고 검수 큐에 남긴다.

## 등록 거래 분류

- `CANCELLED`: 취소 처리, 등록취소, 결제취소 등은 패키지 총계와 등록-입금 미매칭에서 제외
- `NON_PACKAGE_ADJUSTMENT`: 교재비, 차액, 추가결제, 회차수 영향없음, 신규 회차 아님 등은 패키지 총계와 중복 패키지 판정에서 제외
- `UNKNOWN_999`: 설명 없는 `회차수=999`만 비정상 회차수 검토 후보로 분류
- `PACKAGE_REGISTRATION`: 위 예외에 해당하지 않는 등록 패키지 후보
